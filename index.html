Slide 2 / 10 — Mistakes
02/10
What most engineers get wrong

3 mistakes that
break this in prod

1

Fraud detection runs too late

Engineers put fraud inside the risk assessment agent — which runs after full financial signal extraction. You're processing a fraudulent application, wasting compute and creating compliance risk.

2

PII flows through the entire pipeline

Raw PAN and Aadhaar numbers stored in state, passed to every agent, logged everywhere. One bug = data breach. RBI fines start at ₹5 crore.

3

Rejections have no explanation

Approvals get explanations. Rejections get "does not meet criteria." RBI mandates explainability for all credit decisions — not just positive ones.

Slide 3 / 10 — State Schema
03/10
Always start here

Design the state
schema first

class LoanState(TypedDict):
  app_id: str      # unique application ID
  pan_token: str      # hashed — never raw PAN
  monthly_income: float    # from salary agent
  dti_ratio: float    # debt-to-income
  fraud_score: float    # 0.0 → 1.0, hard stop if >0.7
  risk_level: str      # LOW / MEDIUM / HIGH
  decision: str      # approve / reject / HIL
  explanation: str      # RBI required for all decisions
  audit_log: list     # immutable — retained 5 years
→ State is the single source of truth. Every agent reads from it, writes to it, passes it forward.

Slide 7 / 10 — Decision + HIL
07/10
The decision layer

Three-way decision.
HIL for borderline.
Always.

score ≥ 75
✓ Approve
Auto-approved. Explanation generated. Disbursement triggered. End-to-end under 10 seconds.
score 55–74
⚠ HIL
Pipeline pauses via interrupt_before. Human underwriter reviews within SLA. Continues after approval.
score < 55
✗ Reject
Auto-rejected. But explanation agent still runs — specific reasons with figures. RBI mandates this.
Why HIL is non-negotiable: Loan approvals are irreversible. A wrong approval costs money. A wrong rejection creates regulatory risk and loses customers. Never auto-decide borderline cases — the 1% of wrong decisions costs more than the 99% you save.
Slide 8 / 10 — RBI Compliance
08/10
RBI compliance

Every decision needs
a plain-language
explanation

Example — rejection explanation generated by agent
Your application was not approved because your debt-to-income ratio of 0.58 exceeds our maximum threshold of 0.45. Additionally, your CIBIL score of 612 is below our minimum requirement of 650. To improve your eligibility: reduce existing EMI obligations and maintain on-time repayments for 6 months before reapplying.
RBI circular requires: specific reasons for credit rejection, not vague language. "Does not meet criteria" is non-compliant. Audit logs with explanations must be retained for 5 years. All three decision paths — approve, HIL, reject — must generate an explanation.

Slide 10 / 10 — Key Lessons + CTA
10/10
Key takeaways

What makes this
production-ready

01
State schema before code. Define TypedDict state first — every agent reads from and writes to it. This is the contract between all agents.
02
PII masking is node #1. Not inside an agent. Not second. The very first node that runs — before any downstream processing.
03
Fraud = hard stop. Exits the pipeline completely. No financial processing on flagged applications. Ever.
04
Parallel fan-out = 4× speed. 4 independent agents simultaneously → aggregate → sequential reasoning. This is what hits P95 < 10s.
05
HIL before irreversible actions. Borderline cases always go to a human. Loan approvals cannot be undone.
